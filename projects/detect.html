Présentation

Ce projet est un petit analyseur d’URL, conçu pour détecter des caractères invisibles, des homographes Unicode et d’autres formes de stéganographie dans les liens. L’objectif n’est pas de proposer un outil parfait ou professionnel, mais plutôt de comprendre comment fonctionnent ces attaques et d’apprendre à les repérer.

Exécution de l’application (local uniquement)

L’application fonctionne entièrement en local, pour deux raisons simples :

  • aucune URL analysée n’est envoyée à un serveur externe ; tout reste sur la machine,
  • cela me permet de tester, modifier et améliorer l’outil sans backend ni hébergement.

Pour la lancer, j’utilise simplement npm run start dans le dossier de l’application.

Fonctionnement général

Le fonctionnement est volontairement simple. L’outil analyse les URL caractère par caractère pour identifier :

  • les caractères invisibles (espaces zéro-largeur, séparateurs, contrôles),
  • les homographes Unicode (caractères ressemblants visuellement),
  • les mélanges de scripts (latin, grec, cyrillique),
  • certains symboles considérés atypiques dans une URL classique.

Lorsqu’un élément suspect est détecté, l’outil renvoie soit une alerte claire, soit un avertissement lorsqu’il y a un doute mais pas nécessairement un danger. L’idée est d’aider l’utilisateur à douter d’un lien avant de cliquer.

Interprétation des résultats

Pendant mes tests, l’outil affiche 33% de précision. Cela peut prêter à confusion : ce chiffre ne signifie pas que l’outil est « mauvais ».

En réalité, la précision baisse parce que l’analyseur génère beaucoup d’avertissements. Ceux-ci ne sont ni des réussites complètes, ni des erreurs : ce sont simplement des zones grises.

  • 0 échec : aucune URL réellement malveillante n’a été manquée.
  • 100% de détection : l’outil identifie systématiquement les caractères suspects.
  • Les avertissements font baisser la précision affichée, mais ils ne représentent pas un défaut majeur.

Dans un projet orienté sécurité, il vaut mieux avoir trop d’avertissements que pas assez d’alertes. Ce comportement est donc normal et même plutôt souhaitable.

Ce que ce projet m’a appris

Ce projet m’a surtout permis de mieux comprendre les risques liés aux caractères invisibles et aux homographes Unicode. J’ai aussi appris à concevoir une interface simple, à structurer une petite application web locale et à tester différents scénarios d’attaque.

Cela reste un projet modeste, mais il m’a beaucoup appris sur la manière dont les utilisateurs peuvent percevoir un lien comme fiable alors qu’il cache en réalité des éléments trompeurs. C’est un bon rappel que, même dans les petites expériences, on peut toujours améliorer sa compréhension de la sécurité.
